## TL;DR

1. **Гэп — в одну строку:** таблица `benchmark_runs` (db.rs:903) пишется через `report_benchmark`→`insert_benchmark_run` (db.rs:757), но **не читается никем**. `route_task.rs`, `features.rs`, `policy/` к ней не обращаются (grep пустой). Контракт построен (M4 ✅), но данные утекают в никуда.
2. **Срез = пост-inference re-rank по бенчмарк-скору**, а НЕ новая фича в 22-мерном векторе. Зеркалит уже существующий паттерн `PREFERRED_AGENT_BOOST = 0.1` в `route_task.rs`. Ноль ретрейна sklearn-дерева, ноль смены `n_features`, ноль миграций.
3. **Объём:** 1 новый reader в `db.rs` (~15 строк) + ~25 строк в `route_task.rs` + 1 golden-тест. Под флагом `ARBITER_BENCH_WEIGHT` (default 0.0 = выключено) — безопасный A/B, без регрессии когда таблица пуста.
4. **Реальный блокер — данные, не код.** `benchmark_runs` сейчас 0 строк. Чтобы провести первый A/B, не воскрешая Maestro: одноразовый seed-скрипт из готовых ATP-результатов (atp-platform горячий, реальные прогоны есть) → `report_benchmark`. Полная петля через `maestro benchmark` требует R-06b M5 (pending) — это отдельный шаг.

## 1. Где сейчас принимается решение (as-is)

```
route_task.rs:
  build candidates (Vec<AgentInfo>)                    # route_task.rs:190
    → build_feature_vector(agent) -> [f64; 22]         # features.rs:123
       (f[9] = agent_success_rate из agent_stats,      # features.rs:174
        default 0.5; источник — report_outcome)
    → evaluate_for_agents(tree, vectors)               # engine.rs:20
       -> Vec<(agent, PredictionResult)> sort by confidence desc
    → + PREFERRED_AGENT_BOOST к confidence              # route_task.rs (const:62)
    → invariant checks + cascade fallback → выбор
```

22-мерный вектор (engine.rs:55-72) целиком про задачу + операционные стат-данные агента. **Бенчмарк-перформанс не отражён ни в одном измерении.** Reranking-хук под рукой — `PREFERRED_AGENT_BOOST` уже корректирует confidence перед выбором.

## 2. Решение и почему не альтернативы

| Вариант | Что менять | Вердикт |
|---------|-----------|---------|
| **B. Пост-inference re-rank confidence** (выбран) | reader + ~25 строк в route_task | ✅ Тонкий, обратимый, наблюдаемый, A/B-флаг |
| A. Новая 23-я фича в дереве | `n_features` 22→23 везде (Rust+JSON+contract), ретрейн sklearn, новые golden | ❌ Не тонкий; трогает контракт модели |
| A′. Подмешать score в f[9] success_rate | conflate двух сигналов в одной фиче | ❌ Мутит семантику, тихая регрессия operational-стат |

Re-rank выигрывает: дерево остаётся чёрным ящиком как есть, бенчмарк входит отдельной, явно настраиваемой осью, эффект логируется в `decision_path`, A/B — флагом среды.

## 3. Конкретные правки

### 3.1. Reader данных — `arbiter-mcp/src/db.rs` (рядом с `get_agent_stats`, ~:409)

> ↳ **SUPERSEDED (R1).** Unscoped `get_latest_benchmark_score(agent_id)` ниже течёт
> через task_type'ы. Актуальная сигнатура — task_type-scoped
> `get_benchmark_score(agent_id, benchmark_id)`, см. plan Task 1.

```rust
/// Latest benchmark score for an agent (most recent ts). None if no run.
/// Score clamped to [0,1] defensively — контракт не гарантирует диапазон
/// (report_benchmark-v1.schema.json: "score": {"type":"number"}).
pub fn get_latest_benchmark_score(&self, agent_id: &str) -> Result<Option<f64>> {
    let row = self.conn.query_row(
        "SELECT score FROM benchmark_runs WHERE agent_id = ?1 \
         ORDER BY ts DESC LIMIT 1",
        [agent_id],
        |r| r.get::<_, f64>(0),
    ).optional()?;                       // использует idx_benchmark_runs_agent_bench_ts (db.rs:920)
    Ok(row.map(|s| s.clamp(0.0, 1.0)))
}
```
> v1: «последний score по агенту». v2 (позже): фильтр `benchmark_id` под `task_type` задачи + EWMA по нескольким прогонам. Для среза — намеренно один прогон.

### 3.2. Re-rank — `arbiter-mcp/src/tools/route_task.rs` (после `evaluate_for_agents`)

> ↳ **SUPERSEDED (R4).** `ranked` — это `Vec<(String, PredictionResult)>`, правим
> `pred.confidence`; audit пишем в `pred.path` (на месте re-rank нет `result`/
> `decision_path` в scope). Рабочий helper `apply_benchmark_rerank` — см. plan Task 2.

```rust
// рядом с PREFERRED_AGENT_BOOST (:62)
const DEFAULT_BENCH_WEIGHT: f64 = 0.0; // OFF by default — opt-in A/B

let bench_weight = std::env::var("ARBITER_BENCH_WEIGHT")
    .ok().and_then(|s| s.parse::<f64>().ok())
    .unwrap_or(DEFAULT_BENCH_WEIGHT);

if bench_weight > 0.0 {
    for (agent_id, pred) in ranked.iter_mut() {
        if let Some(score) = db.get_latest_benchmark_score(agent_id)? {
            // центрируем на 0.5: нет данных / средний агент = нейтрально (0 сдвиг)
            let delta = bench_weight * (score - 0.5);
            pred.confidence = (pred.confidence + delta).clamp(0.0, 1.0);
            decision_path.push(format!(
                "bench_adjust[{agent_id}]: score={score:.3} Δ={delta:+.3}"));
        }
    }
    ranked.sort_by(|a, b| b.1.confidence
        .partial_cmp(&a.1.confidence).unwrap_or(std::cmp::Ordering::Equal));
}
```
> Идемпотентно к пустой таблице: нет строк → нет сдвига → поведение байт-в-байт как сегодня. Сдвиг центрирован на 0.5 (как default success_rate), поэтому «нет бенчмарка» = нейтрально, а не штраф.

### 3.3. Наблюдаемость
`bench_adjust[...]` уже попадает в `decision_path` → сериализуется в `decisions.decision_path` (db.rs:864) и в obs-span. Дополнительно: одно поле `bench_weight_applied: f64` в `RouteResult` для A/B-разбора.

> ↳ **SUPERSEDED (R3).** Поле `bench_weight_applied` в `RouteResult` **не добавляем** —
> это касание frozen DTO (`861534e`, вендорнут Maestro). A/B-разбор делаем по
> `decision_path` (`bench_adjust[...]`) и obs-span. Контракт не трогаем.

Severity: 🔴 блокер вывода · 🟠 поправить до A/B · 🟡 учесть.

### 🔴 R1 — к §4 и «Рекомендуемым действиям» п.4: тонкий срез даёт ложный kill-signal

Это главное. v1-reader (§3.1) берёт «последний score по агенту» — **один глобальный
скаляр на агента**, без связи `benchmark_id ↔ task_type`. Значит re-rank кладёт
**равномерный сдвиг агента на ВСЕ задачи** — фактически статичный per-agent bias
«агент X вообще лучше, поднять везде». Но ценность benchmark-aware routing именно
в **task-dependent** выборе (codex силён в одном классе задач, aider в другом).

Следствие: если равномерный сдвиг «не двигает маршрут осмысленно», это **не
опровергает** гипотезу benchmark-aware routing — опровергает только «глобальный
bias». А §4 и п.4 предлагают трактовать это как «честный kill-signal для всего
направления». Эксперимент не покрывает то, что собирается убить.

**Фикс:** затащить фильтр `benchmark_id` под `task_type` задачи **в v1, а не в v2**.
Это +~10 строк в reader (WHERE benchmark_id IN (<benchmarks для task_type>)),
не отдельная веха. Без этого A/B невалиден как тест направления — и kill-решение
по нему принимать нельзя.

### 🟠 R2 — к §3.2: центр `0.5` и магнитуда веса произвольны относительно реальных скоров

ATP-скоры (pass@1, % resolved) обычно кучкуются в узкой полосе (напр. 0.3–0.5).
Тогда `(score − 0.5)` почти всегда ≤ 0 → seeded-агенты **штрафуются** относительно
no-data агентов (центрированных в 0). При `weight=0.15`, scores∈[0.3,0.5] →
`Δ∈[−0.03, 0]`, тогда как `PREFERRED_AGENT_BOOST = 0.1`. Бенчмарк-эффект тонет в
шуме boost'а, и «нет сдвига» снова становится артефактом калибровки, а не сигналом.

**Фикс:** центрировать на **наблюдаемом среднем** скоров (или min-max нормировать
в [0,1] по сидам), а не на магической 0.5. Перед A/B залогировать распределение
сид-скоров и подобрать `weight` так, чтобы `|Δ|` был сопоставим с `PREFERRED_AGENT_BOOST`.

### 🟠 R3 — к §3.3: `bench_weight_applied` в `RouteResult` ломает заморозку DTO

§6 (scope guard) обещает «без смены контрактов», но §3.3 добавляет поле в response.
`TODO.md:19`: «Любое изменение API из E2E smoke test → согласование с Maestro +
bump версии». Additive optional field, вероятно, backward-compatible, но формально
это касание frozen DTO (`861534e`, вендорнут Maestro).

**Фикс:** держать A/B-метрику **вне DTO** — только в `decision_path` (уже сериализуется
в `decisions.decision_path`) и в obs-span. Этого достаточно для A/B-разбора по логам,
и заморозка не нарушается. Если поле в DTO реально нужно — пометить как «требует
coordinated release», не прятать под «scope guard: без смены контрактов».

### 🟡 R4 — к §3.2: сниппет не скомпилируется как есть (нет `decision_path` в scope)

На шаге 6 (route_task.rs ~:275) единого мутабельного `decision_path` в scope нет —
путь берётся позже из `prediction.path.clone()` выбранного агента (route_task.rs:348).
`decision_path.push(...)` как написано не соберётся. Решаемо (писать в `pred.path`
каждой записи `ranked`), но оценка «~25 строк» слегка оптимистична.

### 🟡 R5 — к §5: golden-тест проверяет механику, но не «осмысленность»

Предложенный golden (два агента, одинаковый фичевектор, разный score) проверяет, что
re-rank срабатывает — это нужный регресс-гард, оставить. Но он не проверяет R1
(task-dependent выбор). Добавить второй кейс: два разных `task_type`, агенты с
инвертированными per-benchmark скорами → при `weight>0` для task_type A выигрывает
один агент, для B — другой. Вот это и есть тест направления.

### Итог ревью

Срез технически грамотный и правильно «тонкий» — как инженерный приём (re-rank вместо
ретрейна дерева) одобряю целиком. Но как **гейт для go/no-go по направлению он
недо-специфицирован** (R1). Минимальная правка, делающая эксперимент валидным:
`benchmark_id↔task_type` в v1 (R1) + калибровка центра/веса (R2) + метрика вне DTO
(R3). Это добавляет ~полвечера к оценке, но без этого вечерний A/B не даёт права на
вывод «закрыть Maestro».
